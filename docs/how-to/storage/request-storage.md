# Request Storage

## Overview

Use this guide when you need storage provisioned or access adjusted for a research project.

This may include:
- requesting access to an existing Research Data Store (RDS) location
- requesting a new shared storage area for a project
- asking for quota changes or storage expansion
- clarifying which storage service is appropriate before requesting access

If you are still deciding what kind of storage you need, start with:
- [Choose the right service](../../start-here/choose-the-right-service.md)
- [Storage service](../../services/storage/index.md)
- [Storage comparison](../../reference/storage/storage-comparison.md)

## Before you request storage

Make sure you can answer the following:

### 1. What are you storing?

Be clear about the type of data:
- active research data
- code and scripts
- working files for analysis
- outputs and results
- documents for collaboration

### 2. How much storage do you need?

Estimate:
- current data volume
- expected growth over the project
- whether data will be transferred in batches or generated over time

Avoid vague requests such as “a large amount of storage”.

### 3. Who needs access?

List:
- the project owner or principal investigator
- named team members
- whether access is individual or shared
- whether anyone outside UCT needs access

### 4. How will the data be used?

Clarify whether the storage is for:
- shared team access
- access from HPC
- collaboration across devices
- ongoing project storage
- temporary working data during computation

### 5. Are there any data handling constraints?

Note any requirements related to:
- sensitive or restricted data
- controlled access
- institutional or funder requirements
- backup or retention expectations

---

## Typical request information

A storage request is easier to process when it includes:

- project name
- project owner or responsible person
- names of users who need access
- required storage size
- intended use
- whether HPC access is required
- whether the space should be shared
- any time constraints

---

## Example request structure

You do not need to use this wording exactly, but the request should cover the same information.

```text
Project: Coastal Climate Modelling 2026
Owner: Dr Jane Example
Users requiring access:
- Jane Example
- Ahmed Patel
- Lindiwe Ndlovu

Storage required:
- 2 TB initial allocation
- expected growth to 5 TB over 12 months

Purpose:
- active project dataset
- shared access for research team
- data needs to be accessible from HPC

Notes:
- no external collaborator access required
- data should remain available for the duration of the project
```

---

## What happens after the request

After a storage request is submitted, the eResearch team may need to confirm:

- which storage service is most appropriate
- who should be granted access
- whether the requested size is reasonable
- whether the storage should be mounted in additional environments

You may be asked for more detail if the request is incomplete.
